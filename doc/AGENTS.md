# BMAD 團隊協作章程：操作手冊與驗收規範 (AGENTS.md)

本文件定義在「數獨全端整合專案」中，各 AI 專家在撰寫操作手冊、系統運維與功能驗收時的核心職責與協作標準。

## 👥 專家角色與操作手冊職責分工

| 專家代號 | 角色名稱 (Role) | 在操作手冊中的核心職責 | 驗收標準 (Acceptance Criteria) |
| :--- | :--- | :--- | :--- |
| **BMM** | 流程管理專家 (BA) | 主責編寫「使用者操作流程與 SOP」，確保普通玩家、資深玩家與管理員的操作動線符合邏輯。 | 手冊必須包含清晰的步驟（Step-by-Step），且無邏輯死角。 |
| **DMS** | 文件管理專家 (PM) | 確認產品邏輯、定義手冊的版本範圍（PRD v3.1），確保手冊功能與現有代碼 100% 同步。 | 負責維護文件的「單一事實來源（SSOT）」，消除冗餘的舊版文件。 |
| **QAS** | 品質保證專家 (QA) | 負責補上「異常情境處理」與「功能驗收步驟」，例如：登入失敗時的提示、求解器死鎖的解鎖驗證。 | 驗收步驟必須包含預期結果（Expected Results）與邊界條件測試。 |
| **OPS** | 運維部署專家 (Ops) | 編寫「後端維運與資料庫管理」章節，包含 phpMyAdmin 的日誌審查、遠端 FTP 部署注意事項。 | 提供明確的伺服器與資料庫安全維護指令，防範機密外洩。 |

---

## 🔄 手冊編寫與驗收工作流 (Workflow)

1. **需求發動 (DMS/PM)**：確認當前系統功能範圍（登入系統、求解器優化、RWD 彈窗）。
2. **流程梳理 (BMM/BA)**：撰寫各角色的詳細操作 SOP，規劃畫面截圖位置。
3. **異常防護 (QAS/QA)**：加入密碼錯誤、可疑操作（is_suspicious）觸發時的系統反饋與驗收案例。
4. **維運補強 (OPS/Ops)**：寫入資料庫實體結構對齊與伺服器日誌監控說明。
