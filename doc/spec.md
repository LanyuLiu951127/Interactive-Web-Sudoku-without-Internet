# 專案規格書 (spec.md)

## 1. 使用者故事 (User Stories)
*   **US01**: 作為一名玩家，我可以選擇「簡單、中等、困難」難度，以便根據我的水平挑戰遊戲。
*   **US02**: 作為一名玩家，我可以在單元格中填入數字，如果我填錯了，系統會自動標紅提醒我。
*   **US03**: 作為一名玩家，我可以使用「筆記」功能，記錄下某個格子可能的數字，幫助我推邏輯。
*   **US04**: 作為一名玩家，即使在搭飛機（沒有網路）的情況下，我也能打開網頁開始一場數獨。
*   **US05**: 作為一名玩家，我可以隨時暫停遊戲，系統會幫我保存進度，讓我下次回來能繼續遊玩。
*   **US06**: 作為一名玩家，當我卡關時，我可以使用「提示」按鈕，系統會以互動式的步驟引導我推導出該格子的答案，幫助我學習數獨邏輯。
*   **US07**: 作為一名玩家，我希望有「復原」與「清除」按鈕，讓我能輕鬆修正錯誤或嘗試不同解法。
*   **US08**: 作為一名玩家，我希望能隨時暫停計時器，且遊戲會主動遮擋題目，讓我能安心休息不用擔心時間流失。
*   **US09**: 作為一名玩家，我不希望每次都要手動存檔，系統應該默默幫我記下所有的筆記、輸入、時間，讓我即使不小心關掉網頁，下次打開也能無縫接軌。
*   **US10**: 作為一名高階玩家，我希望有「設定」選項可以自由關閉錯誤紅字標示、或是打開「隱藏已完成的數字鍵」等輔助功能，讓我自定義挑戰難度。
*   **US11**: 作為一名新手玩家，我希望能有一個內建的「規則」目錄，讓我可以在遊戲中隨時查詢各種數獨解題技巧，且不會跳轉到其他頁面。
*   **US12**: 作為一名數獨愛好者，當我在實體報紙或其他地方遇到解不開的題目時，我可以使用「數獨求解器」輸入盤面，讓系統幫我快速找出完整答案，作為驗證或參考。

## 2. API 路由定義
雖然核心邏輯在前端，但為了擴充性（如：每日挑戰），定義以下模擬或實際 API：

### `GET /api/game`
*   **功能**: 取得一個隨機生成的數獨題目。
*   **查詢參數**:
    *   `difficulty`: `easy` | `medium` | `hard` (預設為 `easy`)
*   **回應範例**:
    ```json
    {
      "id": "game-12345",
      "difficulty": "medium",
      "board": [
        [5, 3, 0, 0, 7, 0, 0, 0, 0],
        [6, 0, 0, 1, 9, 5, 0, 0, 0],
        // ... 共 9 橫列
      ],
      "solution": [
        [5, 3, 4, 6, 7, 8, 9, 1, 2],
        // ... 完整解答
      ]
    }
    ```

### `POST /api/solve`
*   **功能**: 接收一個由玩家輸入的不完整盤面，並回傳計算後的解答。
*   **要求內容 (Body)**:
    ```json
    {
      "puzzle": [[0,0,0,...], ...] // 9x9 的二維陣列
    }
    ```
*   **回應範例**:
    ```json
    {
      "status": "success",
      "data": {
        "solution": [[5,3,4,...], ...] // 計算出的完整解答
      }
    }
    ```

## 3. 挖洞邏輯定義 (Difficulty Logic)
題目生成後，根據難度隨機移除數字，並確保結果為唯一解：

| 難度 | 剩餘數字量 (提示數) | 挖洞策略描述 |
| :--- | :--- | :--- |
| **簡單 (Easy)** | 40 - 45 個 | 確保每個宮格至少有 4 個數字，均勻分佈，解題路徑明顯。 |
| **中等 (Medium)** | 30 - 35 個 | 隨機挖洞，移除部分宮格的關鍵線索，需要基礎邏輯推理。 |
| **困難 (Hard)** | 22 - 28 個 | 移除大量數字，甚至可能出現某個宮格僅剩 1-2 個提示數，需進階技巧（如 X-Wing）。 |

**驗證標準**: 所有難度生成的盤面，透過「回溯演算法」求解時，必須保證只有一個合法的解。
